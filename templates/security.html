Security SettingsGuide

Fail2ban Configuration Items

  • Trigger Threshold (hits)

    Within the tracking window, an IP reaching this many pattern hits is auto-banned. Default: 5 hits. Lower = more sensitive (risk of false positives); higher = more lenient. See Security Defense page for pattern details.

  • Tracking Window (seconds)

    Time range for accumulating violation counts. Default: 60 seconds. Works with trigger threshold: e.g., default means "5 violations in 60 seconds triggers ban". Larger window makes it easier to trigger.

  • Ban Duration (seconds)

    How long an IP is blocked after first violation triggers a ban. Default: 600 seconds (10 minutes). Fail2ban auto-releases on expiry, but if the IP re-violates within 24 hours, it escalates to a 24-hour ban.

Whitelist Mechanism

  • Official Whitelist

    Panel auto-fetches the latest IP ranges from the following official sources and adds them to the Fail2ban whitelist to prevent blocking legitimate crawlers and CDN nodes:

    • Cloudflare — global CDN and reverse proxy IP ranges (IPv4 + IPv6)
    • Googlebot — Google search crawler official IP ranges
    • Bingbot — Bing search crawler official IP ranges

    When "Auto-update weekly" is enabled, the system re-fetches the latest IPs every Monday at 4:00 AM.

  • Custom Whitelist

    Manually add IPs or IP ranges that should never be banned, one per line. Supports IPv4/IPv6 addresses and CIDR format (e.g. 10.0.0.0/8）。Custom Whitelistdoes NOToverwritten by official fetches。

  • WordPress Security Log Path Whitelist

    Excludes normal files from search engines, ad platforms, site ownership verification, etc. This setting only affects WordPress anomaly access statistics, not generic PHP sites, and does not directly allow or block requests.

Rate Limiting Details

  • How It Works: Token Bucket Algorithm

    The system maintains one token bucket per IP, with capacity equal to the burst buffer, refilling 1 token per second (based on request rate). Requests consume tokens; when depleted, returns 429. Human browsing has natural pauses (reading pages), during which the bucket auto-refills; bots making continuous requests are limited to 1 request/sec once tokens are exhausted.

  • Request Rate (req/min)

    Controls the long-term sustained rate, i.e., token refill speed. Default: 60 req/min (= 1 req/sec). Bot scans usually far exceed this, depleting the token bucket within seconds — nearly all requests return 429.

  • Burst Buffer (hits)

    Controls token bucket capacity, i.e., maximum concurrent requests allowed at page open. Default: 300 hits, covering Elementor + multi-plugin sites (100+ requests/page) with 2-3 tabs open. For more concurrent tabs, set a larger value (e.g., 500-600). Non-Elementor / light sites can lower to 30-120.

    Increasing this valuedoes NOTdelay bot blocking — bots are suppressed by the sustained rate (1 req/sec), independent of burst buffer size. Security is not weakened.

  • ⚠️ Warning: IPs that repeatedly return 429 will be auto-banned by Fail2ban

    429 responses are captured by Fail2ban rules and counted as violations (same level as wp-login brute force). Accumulating to the threshold triggers a firewall ban. Ensure the burst buffer is sufficient for normal use to avoid legitimate visitors accidentally triggering 429 and being blocked.

  • ✓ Increasing burst buffer does not affect bot blocking

    Bots are characterized by continuous requests (no human browsing pauses) and are strictly limited to 1 req/sec by the request rate; after token depletion, nearly all requests return 429. Increasing burst buffer only gives human visitors more leeway for initial page load, security protection is completely unaffected. The real constraint comes from the sustained rate, not the burst ceiling.

  • How to Test Configuration Suitability

    Open the site in a browser incognito window and browse 5-10 random pages (homepage, category, post). Open Developer Tools (F12) → Network tab and check for red 429 status codes. All 200/304 means the configuration is appropriate.

Bot Rate Limiting Guide

  • Scope

    Bot Rate LimitingOff by default. When enabled, each site has its ownBot Rate Limitingbucket；Common bots hitting the same site Bot UA share that bucket，Regular browsersdoes NOTenter this rate limit layer。

  • Search Engine Exemption

    Googlebot/Bingbot are only exempt from bot rate limiting when their source IP matches official IP ranges. Requests impersonating search engines enter the site's bot rate limit bucket.

  • Default Bot Detection

    Currently matches Googlebot, Bingbot, Facebook/Meta (facebookexternalhit, facebook, meta-externalagent), Twitterbot, LinkedInBot, Slackbot, Discordbot, TelegramBot, SemrushBot, AhrefsBot, MJ12Bot, DotBot, and requests with bot, crawler, spider, scraper, or scan in the UA.

  • Recommended Defaults

    Default 30 req/min, burst 20, primarily used to suppress multi-IP fake crawler scanning and reduce WordPress dynamic 404 pressure. After adjusting, monitor 429 counts and normal crawl behavior in access logs.
